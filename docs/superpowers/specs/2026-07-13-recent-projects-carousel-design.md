# Recent Projects Carousel

## Problem

`index.html`'s "Recent Projects" section shows 6 project cards in a static
CSS grid (`repeat(auto-fit, minmax(160px,1fr))`). The request is to turn
this into an animated, swipeable carousel showing 4 cards at a time.

## Scope

- `static-site/index.html` — Recent Projects section markup only.
- `static-site/script.js` — new carousel behavior.
- `static-site/styles.css` — new carousel layout/animation classes.

No other page references this section (`Recent Project` only matches in
`index.html`).

## Design

### Cards per view (pages)

- Desktop (≥1000px): 4 cards per page → 2 pages (6 cards ÷ 4, last page
  partially filled with 2 cards).
- Tablet (700–999px): 2 cards per page → 3 pages.
- Mobile (<700px): 1 card per page → 6 pages.

Breakpoints reuse the pattern already established by other responsive
breakpoints in `styles.css` (checked before adding new ones).

### Markup structure

Replace the current grid `<div>` with:

```html
<div class="jm-proj-carousel" data-proj-carousel>
  <div class="jm-proj-track" data-proj-track>
    <!-- existing 6 .jm-proj-card elements, unchanged -->
  </div>
</div>
<div class="jm-proj-dots" data-proj-dots></div>
```

- `.jm-proj-carousel` is the `overflow:hidden` viewport.
- `.jm-proj-track` is the flex row that gets `transform: translateX(...)`.
- Dots are generated by JS (count = number of pages, recalculated on
  resize) rather than hardcoded, since page count changes per breakpoint.

### Behavior (script.js)

New guarded `DOMContentLoaded` block, following the same structure as the
existing hero carousel (`[data-carousel]`):

1. On load and on resize (debounced), compute `cardsPerView` from
   `window.innerWidth` against the three breakpoints above, then compute
   `pageCount = Math.ceil(cardCount / cardsPerView)`.
2. Render dots (one per page); clicking a dot moves the track to that
   page via `transform: translateX(-100% * pageIndex)` (percentage based
   on viewport width, not fixed px, so it stays correct across
   breakpoints).
3. Touch swipe (`touchstart`/`touchmove`/`touchend`) and desktop
   click-drag (`pointerdown`/`pointermove`/`pointerup`) both compute
   horizontal drag distance; if it exceeds a threshold (~50px), advance
   or retreat one page, clamped to `[0, pageCount - 1]`. Below threshold,
   snap back to the current page.
4. No autoplay. The carousel only moves on explicit user input (swipe,
   drag, or dot click).
5. `prefers-reduced-motion: reduce` → page changes apply instantly
   (no transition), matching the convention already used for the hero
   carousel, scroll-reveal, and count-up animations.
6. No-JS / crawler fallback: without `data-proj-carousel` behavior
   attaching, `.jm-proj-track` still renders as a wrapping flex/grid row
   via CSS fallback so all 6 cards remain visible and readable. This
   mirrors the existing site convention (see `.jm-reveal` and
   `.jm-count` in `CLAUDE.md`) of visible-by-default content that only
   becomes interactive when JS actually runs.

### Styling (styles.css)

- `.jm-proj-carousel` — `overflow:hidden; position:relative;`
- `.jm-proj-track` — `display:flex; transition:transform .4s ease;` each
  `.jm-proj-card` gets a `flex:0 0 (100%/cardsPerView)` set inline by JS
  per breakpoint (since it must match the JS-computed `cardsPerView`).
- `.jm-proj-dots` — small dot row, visually consistent with the existing
  hero carousel's dot styling (reuse/extend those classes rather than
  inventing new visual language).
- `@media (prefers-reduced-motion: reduce)` — `.jm-proj-track` gets
  `transition:none`.

### Out of scope

- No arrow buttons (dots + swipe only, per approved design).
- No autoplay.
- Card content/images/copy unchanged.
- No changes to any page other than `index.html`.

## Testing

No test suite exists for this static site (per `CLAUDE.md`). Verification
will be manual via Playwright screenshots at desktop/tablet/mobile
viewport widths, checking:
- Correct cards-per-view at each breakpoint.
- Dot navigation moves the track correctly.
- Swipe/drag gesture advances/retreats a page.
- `prefers-reduced-motion` disables the transition.
- With JS disabled, all 6 cards are still visible.
